## 3. Per-iteration deep dive

### iter11 (pre-session, on local main as `f5dac2b`)

**Fix**: `bytes_used = end_marker_offset + 8` (not just 4). The attribute end marker is `0xFFFFFFFF` (4 bytes type) + `0x00000000` (4 bytes length), totalling 8 bytes — not 4.

**Symptom this fixed**: `chkdsk` reported `First free byte offset corrected` on every system record because our `bytes_used` was 4 short.

**Spec basis**: MS-FSCC `_FILE_RECORD_SEGMENT_HEADER.FirstFreeByte` MUST point past the 4-byte length following the end-marker type code. The buffer is zero-init so we only need to write the type code, but the count must include the trailing 4 bytes.

**Evidence**: CI iter9 byte-diff (rec 0): reference `bytes_used = 0x210`, ours `= 0x17C` — difference 0x94, but the actual byte-by-byte diff showed the layout matched up to a 4-byte gap at the end-marker.

**Code site**: `src/mkfs.rs::build_system_record` cursor advance after writing the end marker — `cursor += 8` (was previously `cursor += 4`).

### iter12 (pre-session, on local main as `ab75545`)

**Fix**: `MFT_RECORD_IS_VIEW_INDEX` (flag bit 0x0008) on `$Secure` (record 9 in our layout).

**Symptom this fixed**: `Flags for file record segment 9 are incorrect`.

**Spec basis** (publicly documented NTFS layout, MS-FSCC `_FILE_RECORD_SEGMENT_HEADER.Flags`):

| Bit | Name | Meaning |
|----:|------|---------|
| 0x0001 | `MFT_RECORD_IN_USE` | Record currently allocated |
| 0x0002 | `MFT_RECORD_IS_DIRECTORY` | Record hosts an `$I30` (`$FILE_NAME`-keyed) index |
| 0x0004 | reserved / "is 4" | Documented in some sources, unused by chkdsk |
| 0x0008 | `MFT_RECORD_IS_VIEW_INDEX` | Record hosts a **named view index** — anything indexing something other than `$FILE_NAME` (e.g. `$Secure`'s `$SDH`/`$SII` indexes hashed-by-SD-content, `$Quota`'s `$O`/`$Q`, `$ObjId`'s `$O`, `$Reparse`'s `$R`) |

`$Secure` is the canonical view-index host (security-descriptor cache backed by `$SDH` indexed by hash + `$SII` indexed by security ID). chkdsk has hardcoded knowledge of the file name `$Secure` and demands the `IS_VIEW_INDEX` bit on its MFT header even when the on-disk view-index attributes are absent (our v1 ships only an empty `$DATA` stub).

**Why the byte-diff didn't show this**: format.com's rec 9 is `$Quota`, not `$Secure` (modern format.com places `$Secure` outside the first 16 records, presumably under `\$Extend\$Secure`). The reference's rec 9 was structurally a different file from ours, so the flag-byte comparison was uninformative. Fix is keyed on the publicly documented layout description of `$Secure`, not on a flag-byte diff.

**Code site**: `src/mkfs.rs::build_system_record`:

```rust
let is_view_index = record_number == rec::SECURE;
let flags: u16 = 0x0001
    | if is_dir { 0x0002 } else { 0x0000 }
    | if is_view_index { 0x0008 } else { 0x0000 };
```

**Per-field diff**:

| Field | reference (root rec 5) | ours pre-iter13 |
|-------|----------------------:|----------------:|
| $I30 attr `content_size` | 1128 bytes | 48 bytes |
| `INDEX_HEADER.entries_offset` | 16 | 16 |

**Root cause**: every system file's `$FILE_NAME` carries `parent_reference = (rec=5, seq=5)` (the root). chkdsk's Stage 2 unindexed-file scan iterates allocated MFT records, follows each record's `$FN.parent_reference` to the parent directory, and checks for the file name in that directory's `$I30`. Records present in the MFT but absent from the parent's `$I30` are reported as orphans. Our root `$I30` was an empty stub (LAST sentinel only), so every system file came up missing.

**Spec citations**:

- MS-FSCC §2.4 "FSCTL_GET_NTFS_FILE_RECORD" — INDEX_ROOT structure layout.
- MS-FSCC `INDEX_ENTRY` definition: 16-byte header (file_reference + entry_length + content_length + flags + reserved) followed by the indexed key (here a `$FILE_NAME` attribute value) padded to 8.
- COLLATION_FILE_NAME (collation rule = 1): case-insensitive UTF-16 code-unit comparison after upcase fold via `$UpCase` table; on tie, shorter prefix loses.

**Implementation** (`src/mkfs.rs`):

1. New struct `SysIndexEntry { rec, seq, name, is_dir, data_alloc, data_real }`.
2. New function `sort_index_entries(&mut [SysIndexEntry])` — sort by ASCII-uppercase UTF-16 (sufficient for pure-ASCII system file names; equivalent to the `$UpCase`-folded order for these inputs).
3. New function `build_index_root_attr_with_entries(attr_id, idx_block_size, nt_time, parent_ref, entries)` — packs entries into a populated INDEX_ROOT attribute. Each entry's `$FILE_NAME` content mirrors the corresponding inline `$FN`'s `parent_ref`, timestamps, data sizes, file_attrs, and namespace.
4. Move root construction (rec 5) to **after** every other system record so `data_alloc` / `data_real` are known for each entry.
5. Caller builds 12 entries (`$MFT, $MFTMirr, $LogFile, $Volume, $AttrDef, root '.', $Bitmap, $Boot, $BadClus, $Secure, $UpCase, $Extend`), sorts, calls the new function.

**Sort order produced** (matches reference's pattern, with `$Quota` → `$Secure` and `$Extend` added per our layout):

```
$AttrDef, $BadClus, $Bitmap, $Boot, $Extend, $LogFile, $MFT, $MFTMirr, $Secure, $UpCase, $Volume, .
```

(`$` = 0x24 sorts before `.` = 0x2E, hence `.` comes last after all `$`-prefixed names.)

### iter15 (this session, commit `4ee3bad`)

**Fix**: Write FILE-magic placeholders into unused MFT slots (records `12..mft_records_capacity-1`).

**Symptom targeted**: same `frs.cxx 60f` assert. Hypothesis: chkdsk's "64 file records processed" output suggests it iterates the entire `$MFT:$DATA`; raw-zero slots beyond record 11 may trigger the assert.

**Diagnostic** — per-record dump of `reference-mft-16recs.bin` slots 12-15:

```
rec 12: 46494c45 3000 0900 0000000000000000 0c000000 4800 0100 30010000 00100000
        FILE       usa@0x30 cnt=9     LSN=0      seq=12 lc=1 attrs@0x48 fl=01 used=0x130 alloc=0x1000
rec 13: 46494c45 ... seq=13 ...
rec 14: 46494c45 ... seq=14 ...
rec 15: 46494c45 ... seq=15 ...
```

Reference's MFT is sized to exactly 16 records (64 KiB total). Slots 12-15 are 304-byte placeholders with `seq=N` (slot number), `flags=0x01` IN_USE — reference treats them as "reserved for future system use" and pre-reserves bits 12-15 in `$MFT:$Bitmap`.

Ours pre-iter15: pre-allocates 64 MFT slots (262144 bytes for default 256 MiB volume), writes only 12 system records (slots 0-11), leaves slots 12-63 as raw zeros. `$MFT:$Bitmap` keeps bits 12+ clear (genuinely free for future user files).

**Per-field diff** (slots 12-15):

| Field | reference | ours pre-iter15 |
|-------|-----------|-----------------|
| FILE magic | present | absent (raw zeros) |
| `seq` | N (slot number) | n/a |
| `flags` | `0x01` IN_USE | n/a |
| `bytes_used` | `0x130` (304) | n/a |
| `$MFT:$Bitmap` bit | set | clear |

**Implementation note** (the divergence is intentional): we don't follow reference's "`IN_USE=1` + bitmap-bit-set" pattern because our `$MFT:$Bitmap` keeps bits 12+ clear (those slots are genuinely free for user files, not pre-reserved as system). Per the publicly documented NTFS layout, **FILE-magic + `IN_USE=0`** is a valid "free MFT slot" representation: chkdsk reads the slot, sees FILE magic with IN_USE clear, and moves on.

**Code site**: `src/mkfs.rs` — after writing the 12 system records, before writing `$MFT` to disk:

```rust
for slot in 12..mft_records_capacity as u32 {
    let mut placeholder = vec![0u8; rs];
    placeholder[0..4].copy_from_slice(FILE_MAGIC);
    placeholder[REC_OFF_USA_OFFSET..REC_OFF_USA_OFFSET + 2]
        .copy_from_slice(&(USA_OFFSET as u16).to_le_bytes());
    let sectors = rs / bps as usize;
    placeholder[REC_OFF_USA_COUNT..REC_OFF_USA_COUNT + 2]
        .copy_from_slice(&((sectors + 1) as u16).to_le_bytes());
    // seq=0 (never allocated), link_count=0
    let attrs_offset = align8(USA_OFFSET + 2 + sectors * 2);
    placeholder[REC_OFF_ATTRS_OFFSET..REC_OFF_ATTRS_OFFSET + 2]
        .copy_from_slice(&(attrs_offset as u16).to_le_bytes());
    // flags = 0 (NOT in_use); $MFT:$Bitmap also says free
    let used = attrs_offset + 8; // header + end marker (4-byte type + 4-byte length)
    placeholder[REC_OFF_BYTES_USED..REC_OFF_BYTES_USED + 4]
        .copy_from_slice(&(used as u32).to_le_bytes());
    placeholder[REC_OFF_BYTES_ALLOCATED..REC_OFF_BYTES_ALLOCATED + 4]
        .copy_from_slice(&(rs as u32).to_le_bytes());
    placeholder[REC_OFF_MFT_REC_NUM..REC_OFF_MFT_REC_NUM + 4]
        .copy_from_slice(&slot.to_le_bytes());

    placeholder[attrs_offset..attrs_offset + 4]
        .copy_from_slice(&ATTR_END_MARKER.to_le_bytes());
    apply_fixup_on_write(&mut placeholder, bps)?;
    place_record(&mut mft_buf, rs, slot, placeholder)?;
}
```

### iter16 (this session, commit `06d53b4`) — biggest content fix

**Fix**: Replace the runtime-generated `$UpCase` table (synthesised from `char::to_uppercase()`) with the canonical NT 3.x table baked in via `include_bytes!("upcase-canonical.bin")`.

**Symptom targeted**: `Read-only chkdsk found bad on-disk uppercase table - using system table` warning, plus the (then-untested) hypothesis that chkdsk uses `$UpCase` for filename collation in the orphan-recovery scan and a mismatched table → confused comparison → `frs.cxx 60f` assert.

**Background — what `$UpCase` is**:

NTFS's `$UpCase` lives at MFT record 10 and contains exactly 65536 LE u16 values (128 KiB). `upcase[c]` is the uppercase form of BMP code point `c`. NTFS uses this table for case-insensitive filename comparison in B+ tree indexes (COLLATION_FILE_NAME). Both the writer and reader in this crate must consult the same table — but **chkdsk also has a built-in copy and verifies the on-disk table matches**.

**Why the old generator was wrong**:

`src/upcase.rs::generate_upcase_table()` previously synthesised the table from Rust's stdlib:

```rust
let upper: u16 = if (0xD800..=0xDFFF).contains(&cp) {
    cp as u16
} else if let Some(c) = char::from_u32(cp) {
    let mapped = c.to_uppercase().next().unwrap_or(c) as u32;
    if mapped <= 0xFFFF { mapped as u16 } else { cp as u16 }
} else { cp as u16 }
```

This produces a **modern Unicode** mapping (Rust's `char::to_uppercase` follows the current Unicode case-folding rules). Microsoft's NT 3.x table is **far less aggressive** — most code points that modern Unicode case-folds, NTFS preserves unchanged.

**327 BMP code points differ** between the two. Examples (first 10):

| Code point | Description | `char::to_uppercase()` | Microsoft NT canonical |
|-----------:|-------------|------------------------:|------------------------:|
| U+00B5 | MICRO SIGN | 0x039C (GREEK CAPITAL MU) | 0x00B5 (preserved) |
| U+00DF | LATIN SMALL SHARP S "ß" | 0x0053 (S) | 0x00DF (preserved) |
| U+0131 | LATIN SMALL DOTLESS I | 0x0049 (I) | 0x0131 (preserved) |
| U+0149 | LATIN SMALL N PRECEDED BY APOSTROPHE | 0x02BC | 0x0149 (preserved) |
| U+017F | LATIN SMALL LONG S | 0x0053 (S) | 0x017F (preserved) |
| U+019B | LATIN SMALL LAMBDA WITH STROKE | 0xA7DC | 0x019B (preserved) |
| U+01C5 | LATIN CAPITAL D WITH SMALL Z + CARON | 0x01C4 | 0x01C5 (preserved) |
| U+01C8 | similar digraph | 0x01C7 | 0x01C8 (preserved) |
| U+01CB | similar digraph | 0x01CA | 0x01CB (preserved) |
| U+01F0 | LATIN SMALL J WITH CARON | 0x004A (J) | 0x01F0 (preserved) |

Last 5 (high BMP):

| Code point | `char::to_uppercase()` | Canonical |
|-----------:|------------------------:|----------:|
| U+FB13 | 0x0544 | 0xFB13 |

**Background — the two $STANDARD_INFORMATION forms** (publicly documented NTFS layout):

### 5.1 `Read-only chkdsk found bad on-disk uppercase table - using system table` (warning)
