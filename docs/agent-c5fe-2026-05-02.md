#### Evidence — byte-diff vs Microsoft `format.com`

Decoded rec 5's `$INDEX_ROOT` body from `ours-mft-16recs.bin` and
`reference-mft-16recs.bin`:

| field                       | reference | ours (pre-fix) |
|-----------------------------|-----------|----------------|
| `$INDEX_ROOT` total length  | 0x488     | 0x50           |
| Index value `total_size`    | 0x458     | 0x20           |
| Number of `$I30` entries    | 11 (every system file + `.` self-link) | 0 (LAST sentinel only) |

Reference's 11 entries (in collation order):

```
@0x020 ref=  4 seq=4   name='$AttrDef'
@0x088 ref=  8 seq=8   name='$BadClus'
@0x0F0 ref=  6 seq=6   name='$Bitmap'
@0x150 ref=  7 seq=7   name='$Boot'
@0x1B0 ref=  2 seq=2   name='$LogFile'
@0x218 ref=  0 seq=1   name='$MFT'
@0x278 ref=  1 seq=1   name='$MFTMirr'
@0x2E0 ref=  9 seq=9   name='$Quota'   (Microsoft format.com layout has $Quota at rec 9, not $Secure)
@0x340 ref= 10 seq=10  name='$UpCase'
@0x3A0 ref=  3 seq=3   name='$Volume'
@0x400 ref=  5 seq=5   name='.'
@0x458 [LAST entry, len=16, flags=0x2]
```

Ours had only the LAST sentinel.

#### Root cause

`build_empty_index_root_attr` was being used unconditionally for rec
5. With every system record's `$FILE_NAME.parent_reference` set to
`(rec=5, seq=5)`, root was claimed as parent but root's own index
didn't list any of them — chkdsk Stage 2 reconcile reports each as
an orphan.

#### Fix

Added two new symbols and one new call site:

```rust
struct RootIndexChild<'a> {
    rec_num: u32,
    seq: u16,
    name: &'a str,
}

fn collate_filename(a: &str, b: &str) -> std::cmp::Ordering { /* ASCII-uppercase UTF-16 cmp */ }

fn build_root_index_root_attr(
    attr_id: u16,
    index_block_size: u32,
    nt_time: u64,
    root_parent_ref: u64,
    children: &[RootIndexChild<'_>],
) -> Vec<u8> { /* ... */ }
```

The 12 system children sorted under NTFS `COLLATION_FILE_NAME` and
emitted as one `INDEX_ENTRY` each. The `$FILE_NAME` copy embedded in
each index entry zeros `allocated_size` / `real_size` /
`file_attributes` — matching reference (only `$MFT` alone in
reference's index entries carries the real `$DATA` size; the
per-record `$FILE_NAME` is authoritative and we already populate it
correctly).

Two helper details that matter:

- **Collation**. NTFS `COLLATION_FILE_NAME` = uppercase UTF-16 then
  byte compare. The on-disk `$UpCase` table governs the case fold
  for arbitrary Unicode. For ASCII names like `$AttrDef`, `$BadClus`,
  `.`, `$MFT`, ASCII uppercase suffices and produces the order
  reference's index uses verbatim.
- **mft_ref encoding**. Per public NTFS layout: low 6 bytes =
  `record_number`, high 2 bytes = `sequence_number`. So an
  index entry for `$MFT` with `seq=1` is `0x0001_0000_0000_0000`.

The `.` self-link is emitted as a regular index entry pointing at rec
5 (the root itself), with `parent_reference` also pointing at rec 5
— NTFS root is its own parent, so `.` is structurally identical to
"a child file named `.` whose parent is rec 5".

#### Public spec citations

- Microsoft MS-FSCC for `INDEX_ENTRY` and `INDEX_ROOT` headers.
- Microsoft NTFS technical reference (Windows Internals) for the
  collation rule index.

The byte-diff against `format.com`'s reference is the corroborating
evidence.

#### Result

The 12 "Detected orphaned file …" Stage 2 messages are GONE across
every variant. Stage 1 + Stage 2 both verify clean. A new internal
chkdsk error surfaces in the post-Stage-2 unindexed-scan phase:

> An unspecified error occurred (6672732e637878 60f).

That hex is `frs.cxx:60f` (file record segment .cxx, line 1551).
Becomes the central ceiling for the rest of the session.

#### Files touched

- `src/mkfs.rs`: +`build_root_index_root_attr`, +`RootIndexChild`,
  +`collate_filename`. Rec 5 build block now passes the 12-child
  list. Hoisted `attrdef_blob` / `attrdef_clusters` and

### iter14-v1 (committed in `3144024`, reverted in `db38500`) — 72-byte SD on rec 5, DACL after owner — broke mount

#### Hypothesis

The iter13 ceiling is at `frs.cxx:60f`. Reference rec 5 carries 248
bytes of embedded `$SECURITY_DESCRIPTOR`; ours has none. Hypothesis:
chkdsk's post-Stage-2 reconnect-scan needs a parent SD to walk
ACL-cross-checks against, and crashes when root has none.

#### Fix attempt

72-byte self-relative SD with this layout:

```
header (20):
  01 00          rev=1, Sbz1=0
  04 80          control = SE_SELF_RELATIVE | SE_DACL_PRESENT
  14 00 00 00    OffsetOwner = 20
  20 00 00 00    OffsetGroup = 32
  00 00 00 00    OffsetSacl  = 0   (no SACL)
  2C 00 00 00    OffsetDacl  = 44
Owner SID @20 (12): S-1-5-18 (NT AUTHORITY\SYSTEM)
Group SID @32 (12): S-1-5-18 (same)
DACL @44 (28): 8-byte ACL header + 1 ACCESS_ALLOWED ACE
                granting Everyone (S-1-1-0) full access (mask=0x001F01FF)
```

Total 72 bytes.

#### Result — Windows rejected the volume at mount

Diag dir `iter-20260502-053623`:

> chkdsk readonly: "Cannot open volume for direct access."
> Get-Volume: FileSystemType=FAT32 (NTFS detection failed)
> Event Log: Ntfs Event ID 55 × 100, "A corruption was discovered..."

Decoded the bytes back via Python: the SD was structurally valid per
MS-DTYP §2.4.6 (revision=1, control=0x8004, valid offsets, valid
SIDs, ACL_REVISION=2, mask=0x001F01FF). Necessary but not sufficient.

#### What we learned

Reference's SD layout puts the **DACL right after the header**
(OffsetDacl=0x14) with **owner/group at the tail** (OffsetOwner=0xCC,
OffsetGroup=0xDC). Mine put them in the opposite order. Both are
formally legal per MS-DTYP §2.4.6 (the offsets are independent), but
ntfs.sys's mount-time invariant check rejects the our-order SD.

Reverted in commit `db38500`. Falls back to iter13's verified state.

### iter15 (commit `54dda31`) — drop `$Extend` at rec 11; cross-apply c6a1's BPB fixes

Three independent spec-justified fixes bundled (the cross-applied two
came from `agent-c6a1-2026-05-02`'s branch — a sibling agent that

#### Fix 1 — rec 11 left free (was: `$Extend` directory)

##### Evidence — reference rec 11 has no `$FILE_NAME`

Decoded rec 11 attribute table:

```
REF rec11  attrs_offset=0x48 flags=0x0001 bytes_used=0x130 seq=11
  @0x48  $STANDARD_INFORMATION  len=0x48
  @0x90  $SECURITY_DESCRIPTOR   len=0x80
  @0x110 $DATA                  len=0x18
  [end-marker @ 0x128]

OURS rec11 attrs_offset=0x48 flags=0x0003 bytes_used=0x168 seq=11
  @0x48  $STANDARD_INFORMATION  len=0x60
  @0xa8  $FILE_NAME             len=0x68 name="$Extend"
  @0x110 $INDEX_ROOT            len=0x50 name='$I30'
  [end-marker @ 0x160]
```

Reference's rec 11:
- `flags=0x0001` (only `MFT_RECORD_IN_USE`; **no** `MFT_RECORD_IS_DIRECTORY`)
- has no `$FILE_NAME` attribute
- carries `$STANDARD_INFORMATION` + `$SECURITY_DESCRIPTOR` + `$DATA`

Ours:
- `flags=0x0003` (IN_USE + IS_DIRECTORY)
- has a `$FILE_NAME` named "$Extend" with `parent_ref=(5,5)`
- carries an empty `$INDEX_ROOT`
- listed in root's `$I30` as `$Extend`

So our v1 was using rec 11 as a `$Extend` directory linked from root.
Reference doesn't.

##### Public-spec context

Per Microsoft NTFS reservations, records 0–10 are well-defined system
files; rec 11 onwards is implementation-defined for system extensions.
`$Extend` itself has a fixed name but its MFT slot is not specified
to be 11. Modern Microsoft format.com places $Extend somewhere ≥12
and uses rec 11 as a non-indexed system blob.

##### Fix

In `format_filesystem`:
- Remove `rec::EXTEND` from the `$MFT:$Bitmap` initial values
  (slot 11 marked NOT in-use).
- Remove the `RootIndexChild` for `$Extend` from rec 5's children
  list (root `$I30` now has 11 entries instead of 12).
- Delete the rec 11 build block entirely. Slot stays zero-filled
  (no FILE magic, IN_USE bit clear).

`rec::EXTEND` constant kept under `#[allow(dead_code)]` for any
future iter that wants to reuse the slot with the right layout.

##### Result

chkdsk Stage 2 now reports 66 index entries (was 68 — lost the
`$Extend` row + one related count). Verdict: still reaches the same
`frs.cxx:60f` ceiling. Not the root cause, but matches reference layout.

#### Fix 2 — backup boot at the LAST 512-byte sector

##### Evidence (cross-applied from agent-c6a1-2026-05-02 commit 80a3d88)

c6a1 hit Event ID 55 at mount on 32 MiB volumes that 256 MiB tolerated.
Per-byte location diff:

| offset                          | reference | ours pre-fix |
|---------------------------------|-----------|--------------|
| start-of-last-cluster (8191×4096) | (zeros)   | boot signature |
| last-sector (NumberSectors×512) | boot sig  | (zeros)        |

mkfs was writing the backup at `backup_boot_lcn * cluster_size` =
start of the last cluster. ntfs.sys probes byte offset
`NumberSectors × bytes_per_sector` (the very last 512-byte sector,
volume_size − 512). At 32 MiB / 4 KiB cluster the misalignment was
7 sectors. ntfs.sys found zeros at the expected location and refused
to mount.

##### Fix

```rust
// was:
dev.write_all_at(backup_boot_lcn * cluster_size as u64, &boot)?;

// now:
let volume_bytes = cluster_count * cluster_size as u64;
let backup_boot_byte_offset = volume_bytes - bytes_per_sector as u64;
dev.write_all_at(backup_boot_byte_offset, &boot)?;
```

Note: writing it at *both* positions also breaks (ntfs.sys saw two
boot copies and rejected the volume — c6a1 commit 2165997).
Last-sector-only is the correct invariant.

##### Result

32 MiB volumes mount cleanly and reach the same ceiling 256 MiB has
been hitting. Diag `iter-20260502-072454`.

#### Fix 3 — BPB `NumberSectors = volume_sectors − 1`

##### Evidence (cross-applied from agent-c6a1-2026-05-02 commit 84a83d7)

Per-byte BPB diff against a 96 MiB Microsoft-format reference:

| offset | field         | reference (96 MiB) | ours (32 MiB pre-fix) |
|-------:|---------------|--------------------|-----------------------|
| 0x28   | NumberSectors | 0x2FEFF (= 196351 = N − 1) | 0x10000 (= 65536 = N) |

Microsoft's convention: `NumberSectors` counts **data** sectors only;
the trailing backup-boot sector is excluded.

##### Fix

```rust
let volume_sectors: u64 = cluster_count * (cluster_size as u64) / bytes_per_sector as u64;
let number_sectors: u64 = volume_sectors - 1;
b[0x28..0x30].copy_from_slice(&number_sectors.to_le_bytes());
```

##### Public-spec context

Microsoft's NTFS BPB is documented in the publicly published NTFS
layout description; `format.com`'s reference output is the operational
ground truth. At ≥256 MiB the off-by-one was tolerated by ntfs.sys;
at 32 MiB it pushed the kernel's expected backup-boot location past
EOV and ntfs.sys refused mount.

#### Combined iter15 result

Diag dirs `iter-20260502-072332` (256 MiB) / `iter-20260502-072454`
(32 MiB):

- 256 MiB: Stage 1+2 clean → `frs.cxx:60f` (unchanged from iter14-v2).
- 32 MiB: same — was previously failing at mount, now reaches ceiling.

So all 17 `mac:format` matrix variants now compress down to the same

              02 00 34 00 02 00 00 00 00 00 14 00 89 00 12 00

##### Decoded canonical 104-byte SD

```
header (20):
  01 00          rev=1, Sbz1=0
  04 80          control = SE_SELF_RELATIVE | SE_DACL_PRESENT
  48 00 00 00    OffsetOwner = 72
  58 00 00 00    OffsetGroup = 88
  00 00 00 00    OffsetSacl  = 0   (none)
  14 00 00 00    OffsetDacl  = 20

DACL @20 (52):
  02 00 34 00    AclRev=2, Sbz1=0, AclSize=52
  02 00 00 00    AceCount=2, Sbz2=0
  ACE 1 (ACCESS_ALLOWED, 20B): SYSTEM (S-1-5-18), mask=0x00120089
  ACE 2 (ACCESS_ALLOWED, 24B): BUILTIN\Administrators (S-1-5-32-544),
                               mask=0x00120089

Owner @72  (16): S-1-5-32-544 (BUILTIN\Administrators)
Group @88  (16): S-1-5-32-544 (BUILTIN\Administrators)
```

Mask `0x00120089` decodes as:
- `0x00100000` SYNCHRONIZE
- `0x00020000` READ_CONTROL
- `0x00000001` FILE_READ_DATA
- `0x00000008` FILE_READ_EA
- `0x00000080` FILE_READ_ATTRIBUTES

Mask `0x0001009F` (the variant used on rec 3/9/11) adds:
- `0x00010000` DELETE
- `0x00000002` FILE_WRITE_DATA
- `0x00000004` FILE_APPEND_DATA
- `0x00000010` FILE_WRITE_EA

i.e. the same access set plus write/delete capability — appropriate
for `$Volume` (writable label/info) and `$Quota`/`$Secure`-class
records.

#### Fix attempt

Added `SYSTEM_SECURITY_DESCRIPTOR: [u8; 104]` constant with the
canonical bytes verbatim. Wired it into `build_system_record` so
**every** system record gets the SD attribute appended right after
`$FILE_NAME` (attribute order 0x10, 0x30, 0x50, then any extras
like 0x80 / 0x90 / 0xB0).

#### Result — Windows rejected the volume at mount

Diag `iter-20260502-073152`:

> chkdsk readonly: "Cannot open volume for direct access."
> Get-Volume:      FileSystemType: Unknown
> Event Log:       99 NTFS Event ID 55 (corruption discovered)

Verified the SD bytes on rec 0/1/5 byte-by-byte: **identical to
reference**. Yet Windows rejected the volume at mount. So the
hypothesis "SD-presence-fixes-frs60f" is *disproved* for our current
layout — adding spec-correct SD bytes alone is not sufficient.

#### What we learned (negative result)

The kernel cross-checks something between:
- the per-record `$SECURITY_DESCRIPTOR` attribute, AND
- `$STANDARD_INFORMATION.security_id` (we always set 0), AND
- `$Secure`'s `$SDS / $SDH / $SII` view-index attributes (we have
  only an empty `$DATA` stub on rec 9).

When all three are consistent (no SD, security_id=0, $Secure stub) the
volume mounts and chkdsk runs to the frs.cxx:60f ceiling. When SD is
added but the other two stay empty, ntfs.sys catches the inconsistency
at mount and rejects the volume.

#### Reverted

`build_system_record`'s SD emission was removed. The two consts
(`ATTR_SECURITY_DESCRIPTOR`, `SYSTEM_SECURITY_DESCRIPTOR`) remain in
the source under `#[allow(dead_code)]` so iter17 can reuse them
without re-deriving the bytes. Pipeline back to iter15 baseline (diag
`iter-20260502-074121`).

#### Public spec citations

- MS-DTYP §2.4.6 SECURITY_DESCRIPTOR — confirms the canonical layout
  (header → optional SACL → optional DACL → optional owner SID →
  optional group SID, with offsets free).
- MS-DTYP §2.4.4.4 ACE structure / §2.4.4.4.1 ACCESS_ALLOWED_ACE.
- MS-DTYP §2.4.2.2 SID — `S-1-5-18` = NT AUTHORITY\SYSTEM,
  `S-1-5-32-544` = BUILTIN\Administrators, `S-1-1-0` = WORLD/Everyone.
- Microsoft access-mask flags (`0x00120089`, `0x001F01FF`,
  `0x0001009F`) per MS-DTYP §2.4.3.

The byte-diff is the corroborating evidence that the SD content is
correct in isolation.

## 4. Other diagnostic discoveries (didn't lead to a fix this session)

### `$STANDARD_INFORMATION` size mismatch

```
rec 0  REF  STD_INFO len=0x48  (NTFS 3.0 layout: 48-byte value)
rec 0 OURS  STD_INFO len=0x60  (NTFS 3.1 layout: 72-byte value)
```

We emit NTFS 3.1 sized `$STANDARD_INFORMATION` (with `max_versions`,
`version_number`, `class_id`, `owner_id`, `security_id`,
`quota_charged`, `update_sequence_number` extension fields). Reference
emits NTFS 3.0 sized (48 bytes — original WinNT 4 fields only). Both
are valid per spec; both volumes declare NTFS 3.1 in `$Volume`.

This is a layout choice not a bug, but worth noting for any future
iteration that wants to compress system records. Switching to 48-byte
`$STANDARD_INFORMATION` saves 24 bytes per record × 11 records =
264 bytes total.

### `next_attr_id` field

```
rec 5  REF  next_attr_id = 0x05  (highest used + 1)
rec 5 OURS  next_attr_id = 0x10  (constant 16)
```

Layout choice, not a bug — the field's only constraint is that future
attribute additions allocate IDs > `next_attr_id`. Our 16 is safe.
Reference is more compact.

### `$UpCase` (Bug B from iter6, still open)

chkdsk emits "Read-only chkdsk found bad on-disk uppercase table -
using system table." every run. Our `src/upcase::generate_upcase_table`
uses Rust's `char::to_uppercase()`, which follows current Unicode case
rules. Microsoft's NTFS uses a fixed historical NT 5.x table that
diverges from current Unicode.

The right fix is to derive the canonical 128 KiB table from
`format.com`'s reference output (or an equivalent publicly published
source) and ship it as a `const` byte array. Defer to a separate
iteration — this is a non-fatal warning that doesn't block the
matrix.

### Backup boot wrote in TWO places — also wrong

Per c6a1's iter14 (commit 2165997): writing the backup boot at *both*
the last sector AND start-of-last-cluster also breaks ntfs.sys
(`Cannot open volume`). Last-sector-only is the correct invariant.
We never had this bug because iter15's first attempt at the fix went
straight to last-sector-only.

### Pre-existing GPL-tooling reference

`src/mkfs.rs:13–15` cites Flatcap URLs in the module docstring. This
violates the project's hard rule (CLAUDE.md / project memory) on
GPL-tooling name-drops. Pre-dates this session; flagged for a
separate cleanup commit.

## 5. iter17+ candidates (open hypotheses, not yet attempted)

The post-Stage-2 `frs.cxx:60f` ceiling has survived: iter13 (root
$I30), iter14-v2 (root SD), iter15 ($Extend drop + BPB fixes),
iter16-attempt (SD on every system rec). Open hypotheses:

1. **`$Secure` view-index attributes (`$SDS / $SDH / $SII`)**.
   Reference rec 9's `$Secure` has these populated; ours has only an
   empty `$DATA` stub. chkdsk's reconnect-scan may dereference them
   when walking each record's security descriptor.

2. **`$STANDARD_INFORMATION.security_id != 0` paired with per-rec SD**.
   We write `security_id = 0` everywhere. With no per-record SD,
   security_id=0 means "use parent's defaults" and ntfs.sys
   tolerates it (iter15 baseline). When SD is added but security_id
   stays 0, the kernel may detect the inconsistency. Test plan: add
   SD on every record AND set `security_id` to 1 (or some non-zero
   value pointing into a valid `$Secure` index entry).

3. **`$LogFile` RSTR records**. We fill `$LogFile` with 0xFF bytes
   and trust ntfs.sys to re-init on first mount. Modern ntfs.sys may
   demand at least one valid RSTR record header at the start of the
   log.

4. **Canonical `$UpCase` bytes**. The reconnect-scan keys on something
   the system fallback table reports differently from the on-disk
   table. Possibly low-priority since it's been a non-fatal warning
   since iter6.

The recommended next step: build out hypothesis 1 + 2 in tandem
(populate `$Secure` view indexes AND set `security_id` to point at a
valid `$Secure` entry AND emit per-record SD using the iter16 const).
That's a coherent triple-fix; any one alone seems to break mount
(iter16-attempt) or have no effect (iter14-v2).

### Update post-merge — agent-8a29's parallel SD attempt also disproved the hypothesis

When this branch merged `main`, three new commits arrived from sibling
agents that this session didn't see in time:

1. **`091848d` `mkfs: $SECURITY_DESCRIPTOR (0x50) on every system MFT
   record`** (agent-8a29-2026-05-02). agent-8a29 implemented the
   SD-on-every-system-record fix as well, with a richer blob set (RO
   for read-only system files, RW for writable, plus a separate
   248-byte ROOT blob with multi-ACE INHERIT_ONLY). Their commit
   message states verbatim:

   > This was an evidence-corroborated fix targeted at chkdsk's
   > "An unspecified error occurred (frs.cxx 60f)" Stage 2 assert.
   > The SD layout is now correct; the assert STILL FIRES. Root
   > cause lies elsewhere — most likely the all-zero MFT slots 12..63
   > (reference fills slots 12..15 with FILE-magic placeholders;
   > ours has zeros).

   So **two independent agents** (c5fe-iter16-attempt + agent-8a29
   `091848d`) tried the SD-everywhere hypothesis and reached the
   same conclusion: SD bytes correct → frs.cxx:60f still fires.
   Strong corroboration that SD is **not** the root cause. The
   difference: agent-8a29 kept their SD code in (it's structurally
   correct + matches reference), while my iter16-attempt was
   reverted to keep the dead-code shelf out of mainline. After the
   merge, main is on agent-8a29's "keep SD in" stance.

2. **`d620205` `mkfs: bake Microsoft canonical NT 3.x $UpCase table
   (replace generator)`**. Replaces the runtime-generated UpCase
   from `chars::to_uppercase()` with a baked 128 KiB const that
   matches Microsoft's NT 3.x table byte-for-byte. Closes Bug B
   from iter6 (chkdsk's "Read-only chkdsk found bad on-disk uppercase
   table - using system table" warning). Doesn't fix the
   frs.cxx:60f assert (already known from agent-8a29's commit
   message), but eliminates the warning.

3. **`7072242` `mkfs: $STANDARD_INFORMATION 48-byte (NTFS 1.x) form
   on system records`**. Switches `$STANDARD_INFORMATION` from
   NTFS 3.1 (72 bytes) to NTFS 1.x / 3.0 (48 bytes) on system
   records, matching `format.com`'s output exactly. This is the
   "size mismatch" I flagged in §4 of this document as a layout
   choice. Now resolved.

### Updated iter17+ candidate (most-likely)

After the merge, hypothesis 1 (`$Secure` view indexes) and hypothesis
2 (`security_id != 0`) remain open. Hypothesis 3 (`$LogFile` RSTR)
and hypothesis 4 (canonical `$UpCase`) are now PARTIALLY resolved
— hypothesis 4 by `d620205` (UpCase baked), hypothesis 3 still open.

**New top candidate, surfaced by agent-8a29's `091848d` commit
message: MFT slots 12..15 should carry FILE-magic placeholder
records.** The reference dump shows all 16 slots have FILE magic
even though only slots 0..11 are user-meaningful system files.
Slots 12..15 are reserved with valid record headers but no
attributes (or only `$STANDARD_INFORMATION`). Our writer leaves
slots 12..15 fully zeroed.

Why this might be the cause of frs.cxx:60f: chkdsk's reconnect-scan
phase walks every MFT slot up to a max counted from `$MFT:$Bitmap`.
Our `$MFT:$Bitmap` lists 11 in-use records; reference's lists 16.
chkdsk may iterate slots 12..15 expecting FILE magic on each, hit
zeros, and assert internally rather than treating them as "free".

Test plan for iter17 (next session):
1. Add slots 12..15 as FILE-magic-only placeholder records in
   `$MFT:$Bitmap` (set those bits) AND write minimal valid records
   to those slots (`FILE` magic, valid USA, `IN_USE=0`, no
   attributes — just the end marker).
2. Re-run the pipeline. Expected: either frs.cxx:60f gone, or a
   different assert surfaces.
