### 2.1. `7701332` mac-format-tiny-32mib: BPB NumberSectors = volume_sectors - 1

**File**: `src/mkfs.rs` (around line 647 of the pre-fix source).

**Diff**:

```rust
-    // Total sectors (volume size in sectors). Includes the very last
-    // sector which contains the backup boot.
-    let total_sectors: u64 = cluster_count * (cluster_size as u64) / bytes_per_sector as u64;
-    b[0x28..0x30].copy_from_slice(&total_sectors.to_le_bytes());
+    // BPB NumberSectors at offset 0x28: count of *data* sectors in the
+    // volume, NOT counting the trailing backup-boot sector. Microsoft
+    // format.com always writes (volume_sectors - 1). Corroborated in
+    // iter13 (agent-840e-2026-05-02): a 96 MiB reference volume showed
+    // NumberSectors=0x0002FEFF for 196352 partition sectors, i.e. N-1.
+    // We previously wrote N (the full sector count); at >= 256 MiB the
+    // off-by-one was tolerated by chkdsk + ntfs.sys, but at 32 MiB it
+    // pushes the kernel's expected backup-boot location past EOV and
+    // ntfs.sys refuses to mount the volume (Get-Volume reports
+    // FileSystemType=Unknown, Size=0).
+    let volume_sectors: u64 = cluster_count * (cluster_size as u64) / bytes_per_sector as u64;
+    let number_sectors: u64 = volume_sectors - 1;
+    b[0x28..0x30].copy_from_slice(&number_sectors.to_le_bytes());
```

**Why this is wrong, what evidence**: byte-diff of our boot sector vs.
Microsoft `format.com`'s on a 96 MiB reference volume showed:

| BPB offset | field | reference (96 MiB) | ours (32 MiB pre-fix) | conclusion |
|------------|-------|--------------------|-----------------------|------------|
| 0x28 (8 B) | NumberSectors | `0x0002FEFF` (= 196351 = N-1 for 196352 partition sectors) | `0x00010000` (= 65536 = N for 32 MiB) | provably wrong on our side |

The pre-existing source comment (`"Includes the very last sector which
contains the backup boot."`) showed the original author was aware that
the very-last-sector hosts a backup-boot copy — but resolved the
question of whether to count it the wrong way. Microsoft's convention
across every reference dump is `N - 1`.

**Public spec citation**: NTFS BPB convention — the field at offset 0x28
is documented in MS-FSCC and reproduced in every public NTFS layout
reference (Windows Internals, Microsoft NTFS technical reference) as
"number of sectors in volume" with the standing convention that the
trailing sector hosts the backup boot copy and is not counted. The byte-
diff against `format.com` corroborates the convention in the only way
that matters for chkdsk acceptance.

**Effect alone**: spec compliance achieved. By itself this fix did NOT
move the chkdsk verdict — at 256 MiB the volume still hit the orphan-
recovery cascade (which my second fix addresses), and at 32 MiB the
volume STILL refused to mount because the orphan-recovery scan trips a
deeper problem before the kernel's small-volume tolerance pathway can
recover.

**Effect in combination with fix 2**: 32 MiB volume now mounts cleanly
under Windows. Pre-iter15-cherry-picks: `FileSystemType: Unknown, Size:
0`. Post: `FileSystemType: NTFS, Size: 33,554,432, FileSystemLabel: TINY`.

### 2.8. `950397a` mkfs: iter14 — write $SECURITY_DESCRIPTOR (0x50) on every system record

**File**: `src/mkfs.rs` (84 lines added: const + helper + insertion in
`build_system_record`).

**What it does**: defines a 104-byte `const SYSTEM_SECURITY_DESCRIPTOR:
[u8; 104]` matching the byte layout Microsoft `format.com` writes on
every system record. Adds a `write_security_descriptor()` helper that
emits a resident attribute (type 0x50, attr_id=2, value=the const). The
helper is called inside `build_system_record` after `write_file_name`
but before the `extra_attrs` loop, so every system record now carries
the 0x50 attribute.

**SD byte layout** (from MS-DTYP §2.4.6 self-relative SECURITY_DESCRIPTOR):

```
@0  header (20 bytes):
       01 00          rev=1, Sbz1=0
       04 80          Control = SE_SELF_RELATIVE | SE_DACL_PRESENT
       48 00 00 00    OffsetOwner = 72
       58 00 00 00    OffsetGroup = 88
       00 00 00 00    OffsetSacl  = 0   (no SACL)
       14 00 00 00    OffsetDacl  = 20

@20 DACL (52 bytes):
    @20  ACL header (8): rev=2, Sbz1=0, AclSize=52=0x34, AceCount=2, Sbz2=0
    @28  ACE[0] (20): ACCESS_ALLOWED for SYSTEM (S-1-5-18), mask=0x00120089
    @48  ACE[1] (24): ACCESS_ALLOWED for BUILTIN\Administrators
                     (S-1-5-32-544), mask=0x00120089

@72 Owner SID (16): S-1-5-32-544 (BUILTIN\Administrators)
@88 Group SID (16): S-1-5-32-544 (same)
```

Mask `0x00120089` = `SYNCHRONIZE | READ_CONTROL | FILE_READ_DATA |
FILE_READ_EA | FILE_READ_ATTRIBUTES`.

**Public spec citation**:

- MS-DTYP §2.4.6 SECURITY_DESCRIPTOR (self-relative form layout).
- MS-DTYP §2.4.5 ACL (DACL header layout).
- MS-DTYP §2.4.4 ACCESS_ALLOWED_ACE (ACE structure).
- MS-DTYP §2.4.2 SID (well-known SIDs S-1-5-18 = SYSTEM,
  S-1-5-32-544 = BUILTIN\Administrators).

**Why this hypothesis**: agent-8934's iter14 (commit `8b8e5a1`)
documented via byte-diff that the reference carries this exact 104-byte
SD on every system record (rec 0..11) and ours carried none. They
recorded the hypothesis but did not implement the fix. Their reasoning:
"chkdsk's 'scanning unindexed files for reconnect' sub-phase walks each
MFT record and validates its `$STANDARD_INFORMATION` security_id field
against `$Secure`'s `$SDH`/`$SII` indexes (or, for system records,
against the per-record `$SECURITY_DESCRIPTOR`). When a record carries
no 0x50 AND its `$STANDARD_INFORMATION` security_id points at an entry
that doesn't exist in `$Secure` (we set it to 0, which is invalid),
chkdsk hits an internal sanity check at frs.cxx 60F."

**Result**:

- 7/7 cargo tests still green.
- Pipeline diag dir: `$TMPDIR/rust-fs-ntfs-diag/agent-840e-2026-05-02/iter14-sd-on-all-records-256mib/`.
- Volume mounts cleanly. chkdsk Stage 1 + Stage 2 still clean.
- **`frs.cxx 60f` chkdsk error PERSISTS** (chkdsk readonly exit 3,
  /scan exit 11 — same as pre-iter14).

**Conclusion**: this **eliminates "missing 0x50 SD on system records"
as the cause of `frs.cxx 60f`.** The SD is structurally correct and
matches the reference byte-for-byte, but the residual error survives.
The hypothesis was sound (the missing 0x50 is a real spec gap and
worth fixing on its own) but is not the proximate cause of the
post-Stage-2 ceiling.

agent-5442's earlier iter14-v2 had reached the same conclusion using
a simplified 72-byte Everyone-full-access SD on root rec 5 only. My
iter14 is the full-strength corroborated version applied to every
system record — same negative result, with stronger evidence (no
ambiguity that "maybe the simplified SD content was the issue").

|-------|-----------|--------------|

Hypothesis: `security_id = 0` is invalid (no entry exists in `$Secure`
for ID 0) and trips chkdsk's `frs.cxx 60F`.

  form is documented as the canonical minimum; the 24-byte 3.x extension

  implementation chooses to write them").
- NTFS layout convention: rec 11 (`$Extend`) is a documented system
  record holding `$Quota`, `$ObjId`, `$Reparse`, `$UsnJrnl` as named-

## 3. What I discovered

### 3.1. `mac-format-tiny-32mib` was failing for **two** independent reasons

Pre-cycle-1 state: 32 MiB volumes formatted by `mkfs_ntfs` were rejected
by Windows with `FileSystemType: Unknown, Size: 0`. chkdsk could not
even open the volume for direct access.

This was diagnosed during cycle 1 as the BPB.NumberSectors off-by-one
(fix 2.1) and proven via byte-diff. But applying just that fix did NOT
resolve the symptom — the volume still showed `FileSystemType: Unknown`.

**The actual resolution required the COMBINATION of fix 2.1 (NumberSectors)
AND fix 2.6 (root `$I30` populated)**. Either fix alone is insufficient
at 32 MiB; together they let ntfs.sys complete its mount path and chkdsk
proceeds through Stage 1 + Stage 2.

This is a useful generalisation: at 256 MiB the volume *was* mounting
even with both bugs present, just hitting the orphan-recovery cascade.
At 32 MiB the kernel's small-volume tolerance is tighter and BOTH bugs
must be absent for mount to succeed.

### 3.2. `frs.cxx 60f` is the universal post-Stage-2 chkdsk ceiling

Across every Mac-format scenario at any volume size ≥ 32 MiB and any
cluster size ≥ 1 KiB, the chkdsk progression is:

```
Stage 1: Examining basic file system structure ...    ✓ 64 file records processed
File verification completed.
0 large file records processed.
0 bad file records processed.
Stage 2: Examining file name linkage ...              ✓ 68 index entries processed (or 66 post-iter14-v3)
Index verification completed.
CHKDSK is scanning unindexed files for reconnect to their original directory.
An unspecified error occurred (6672732e637878 60f).
```

Decoding the magic number: `6672 732e 6378 78` is the ASCII string
`frs.cxx`. So chkdsk is leaking its source file pointer + line offset
0x60f = 1551 into the error formatter when its internal `Assert()` fires
during the orphan-recovery scan.

NTFS Event ID 55 also fires during this run (~100 entries in the kernel
log), corresponding to ntfs.sys reporting "corruption discovered" each
time chkdsk's `Assert()` trips.

**Hypotheses tested and ELIMINATED for `frs.cxx 60f`**:

| # | Hypothesis | Tested by | Verdict |
|---|------------|-----------|---------|
| 1 | Missing 0x50 `$SECURITY_DESCRIPTOR` on system records | this session iter14 (`950397a`) | doesn't fix — full reference-corroborated 104-byte SD on every record, no change to verdict |
| 2 | `$STANDARD_INFORMATION` 3.x form with `security_id=0` | this session iter15 (folded into `6ecf58c`) | doesn't fix — shrunk to 48-byte 1.x form, no change |
| 3 | rec 11 (`$Extend`) present as a directory with parent-ref to root | agent-5442 iter14-v3 (`6ecf58c`) | doesn't fix — rec 11 fully removed, no change |
| 4 | rec 11 (`$Extend`) present with empty `$I30` (early agent-c6a1 baseline) | implicit baseline | doesn't fix — pre-existing state |

**Standing hypothesis (untested by my session)**: `$Secure`'s
`$SDH` / `$SII` view-index attributes plus `$SDS` data stream content.
Currently we ship `$Secure` (rec 9) as `0x10/0x30/0x80` with an empty
12-byte resident `$DATA` and the `MFT_RECORD_IS_VIEW_INDEX` flag set
in the record header (per agent-c6a1's iter12 fix). Microsoft's
reference ships:

- A populated `$SDS` data stream containing at least one canonical
  security descriptor entry (the BUILTIN\Administrators read-only SD
  shown in fix 2.8 above).
- A named `$SDH` index attribute (`$INDEX_ROOT` + `$INDEX_ALLOCATION`
  + `$BITMAP` named "$SDH") keyed on the security-descriptor hash.
- A named `$SII` index attribute (same triple, named "$SII") keyed on
  the security-id integer.

The chkdsk `frs.cxx 60F` assertion is in code that walks the
`$STANDARD_INFORMATION.security_id` of every record and looks it up via
`$SII` to recover the actual SD. With our `$SII` empty (or absent
entirely on rec 9), the lookup fails — but we now write 0x50 directly
on every system record, so the lookup *should* be skipped. The fact
that it isn't suggests chkdsk's logic is "first try `$SII` lookup, if
that fails fall back to the per-record 0x50" — and the failure path
itself is buggy in ways that fire `Assert()`.

This is consistent with `frs.cxx 60F` being a chkdsk *internal*
defensive assertion that fires when `$Secure`'s view indexes don't
exist at all (regardless of whether per-record 0x50s are valid). Fixing
it requires implementing the full `$Secure` view-index machinery —
substantially deeper work, agent-5442 estimated ~300 lines of new mkfs
code in their iter14-v2 closing note.

### 3.3. Cluster-size scenarios surface a separate "Corrupt MFT" failure class at < 4 KiB clusters

`mac-format-cluster-512` (512-byte clusters) and
`mac-format-cluster-1k` (1024-byte clusters) — both at 256 MiB volume
size — fail differently from the universal `frs.cxx 60f` ceiling:

```
Corrupt master file table. Windows cannot recover MFT. CHKDSK aborted.
```

chkdsk's first MFT read fails outright. Format succeeds, mount succeeds,
but chkdsk's MFT walk trips immediately.

**Hypothesis (per agent-8934 iter15, untested)**: when `cluster_size <
mft_record_size`, the BPB encoding for `clusters_per_mft_record`
becomes negative-power-of-2 (representing the mft_record_size as
`2^|cpmr|` bytes). MFT record placement code that assumes ≥1 cluster
per record may compute incorrect record offsets. At cluster_size 4096
(default mft_record_size), cpmr = +1; at 8192, cpmr = +2; at 64 KiB,
cpmr = +16. At 1024-byte clusters, cpmr = -12 (= 2^12 = 4096 bytes per
record); at 512-byte clusters, cpmr = -12 (same, since
mft_record_size=4096 either way).

Inspect this by walking `src/mkfs.rs`'s `build_boot_sector` cpmr
calculation against the MFT placement code in `format_filesystem`. The
boot-sector code (lines around 654-664) handles both signs correctly;
the placement code may not.

This is **distinct** from `frs.cxx 60f` and survives all four hypotheses
tested above. agent-c5fe-2026-05-02 marked these as
`passed-to-frs60f-ceiling` in the work-list (note: their cycle-3 run
appears to have lost the cluster-lt-4k distinction; agent-8934's
earlier marks captured it correctly).

## 7. What's still broken, in priority order

### 7.1. `frs.cxx 60f` chkdsk residual (universal)

Affects all 12 Class B scenarios. Standing hypothesis: implement
`$Secure`'s `$SDH` / `$SII` view-index attributes plus a populated
`$SDS` data stream. Concretely:

- Build a `$SDS` blob containing at minimum one canonical SD entry
  (the BUILTIN\Administrators read-only one shown in fix 2.8 above)
  plus the surrounding `SECURITY_DESCRIPTOR_HEADER` (hash, security_id,
  offset_in_$SDS, size).
- Build a named `$INDEX_ROOT` + `$INDEX_ALLOCATION` + `$BITMAP` triple
  for `$SDH` (indexed on the SD's hash) — root has the END sentinel
  unless we want to index the one entry.
- Same triple for `$SII` (indexed on security_id integer).
- Update every record's `$STANDARD_INFORMATION` `security_id` to point
  at the entry we wrote in `$SDS` (this requires either reverting fix
  2.9 to the 72-byte 3.x form, or accepting that 48-byte 1.x records
  use the per-record 0x50 path entirely).

agent-5442's iter14-v2 estimate: ~300 lines of new mkfs code. I did
not attempt this.

### 7.2. Cluster-lt-4k corrupt MFT (cluster-512, cluster-1k)

Distinct from `frs.cxx 60f`. Hypothesis: MFT record placement in
`format_filesystem` assumes ≥1 cluster per record. Re-investigate
against `src/mkfs.rs::build_boot_sector`'s cpmr (clusters-per-MFT-record)
computation and the MFT placement loop. Likely a small fix once
identified.

### 7.3. Win-fixture runner not implemented (5 scenarios)

The `WinFixtures` parameter is plumbed into `run-windows-test.ps1`
(my fix 2.3) but no PowerShell code reads it and creates the named
fixture files on the mounted volume between mount and chkdsk. Adding it
is straightforward — a `foreach ($f in $fixtures) { Set-Content ... }`
loop after the drive-letter assignment, before chkdsk runs.

### 7.4. Win-format inverted runner (3 scenarios)

The current `run-windows-test.ps1` runs our mkfs first and uses
Microsoft `format.com` only as a parallel byte-diff reference. The
`win-format-*` scenarios need format.com to be the only formatter, with
our mkfs/reader providing the verification leg. New PowerShell file
or a `-Mode WinFormat` switch on the existing one.

### 7.5. Mac-side writer / deleter CLI (1 scenario each)

`fs_ntfs::facade::Filesystem::create_file` and `Filesystem::unlink` /
