VM via a GPT-partitioned VHDX wrapper. The volume is small enough to

### Reference

the same pipeline. The two volumes' first 16 MFT records dump into
`reference-mft-16recs.bin` and `ours-mft-16recs.bin`. Per-record
byte-diff is the corroboration mechanism.

### Pipeline

`scripts/test-windows-local.sh` (Mac side) tar-ssh's the source onto
the VM, runs `scripts/run-windows-test.ps1` (Windows side) which

streams `diag/` back to `${TMPDIR}/rust-fs-ntfs-diag/<session>/iter-<ts>/`
on the Mac. ~30–90 s per cycle.

## Public references used

Every byte that this session touched cites at least one of these. No
GPL'd Linux NTFS reimplementation was consulted; the protocol's
hard rule prohibits it and the project memory entry
`feedback_no_gpl_tooling_references` reinforces it.

- **MS-FSCC** — File System Control Codes spec, Microsoft-published.
  Defines `_FILE_RECORD_SEGMENT_HEADER` (the MFT-record header at
  offset 0..0x30+), `$STANDARD_INFORMATION` value layout, `$FILE_NAME`
  attribute layout, `$INDEX_ROOT` / `$INDEX_HEADER` /
  `INDEX_ENTRY` shapes, the `MFT_RECORD_*` flags
  (`IN_USE`/`IS_DIRECTORY`/`IS_VIEW_INDEX`).
- **MS-DTYP** — Windows Data Types spec. Defines
  `SECURITY_DESCRIPTOR`, `ACL`/`ACE`, `SID`. Used by sibling agents
  for iter14's `$50` work; this session did NOT add SD generation.
- **Windows Internals** (Russinovich/Solomon) — qualitative reference
  for what each NTFS system file does. Used for understanding which
  records chkdsk treats as special and why `$Secure`/`$Quota`/`$Extend`
  matter.
- **The diag dump itself** — for fields where the public spec is
  silent or ambiguous (e.g. exactly which entries Microsoft puts in
  root's `$I30`, exactly what `frs.cxx:60f` means), reference behavior
  observed on the wire is the citation. Per `corroborated-debug`'s
  Phase 4: "If you can't find a public source for a field's meaning,
  prefer 'we observed reference does X; chkdsk reports Y when we
  don't' as the citation."

> Detected orphaned file $Boot (7), should be recovered into directory file 5.

Verbatim from
`$TMPDIR/rust-fs-ntfs-diag/agent-5442-2026-05-02/iter-20260502-024032/chkdsk-readonly.txt`.

Each system record's `$FILE_NAME` declares `parent_reference =
(rec=5, seq=5)`. chkdsk Stage 2 walks the parent's `$I30` looking
for entries that point back at each child; when it finds none it
reports the child as "orphaned".

#### Diagnostic — byte-diff of root rec 5's `$INDEX_ROOT`

Decoded `$INDEX_ROOT '$I30'` from rec 5 in both
`reference-mft-16recs.bin` and `ours-mft-16recs.bin` of the same
diag dir. Walked attribute headers from
`MFT_RECORD.attrs_offset` (0x48 in our 4096-byte records),
identified the type-0x90 attribute named `$I30`, decoded its
resident value (16-byte IR header + 16-byte INDEX_HEADER +
sequence of `INDEX_ENTRY`s).

| Field                       | reference         | ours pre-iter13 |
|-----------------------------|-------------------|-----------------|
| `$INDEX_ROOT` total length  | 0x488             | 0x50            |
| INDEX_ROOT value size       | 0x468             | 0x30            |
| INDEX_HEADER.entries_used   | 0x458             | 0x20            |
| INDEX_ENTRY count           | 12 + LAST sentinel | LAST sentinel only |

Per-entry decode of the reference's `$I30` (sorted by NTFS
`COLLATION_FILE_NAME`):

| Entry | (rec, seq) | e_len | s_len | name      |
|-------|------------|-------|-------|-----------|
| 0     | (4, 4)     | 0x68  | 0x52  | $AttrDef  |
| 1     | (8, 8)     | 0x68  | 0x52  | $BadClus  |
| 2     | (6, 6)     | 0x60  | 0x50  | $Bitmap   |
| 3     | (7, 7)     | 0x60  | 0x4c  | $Boot     |
| 4     | (2, 2)     | 0x68  | 0x52  | $LogFile  |
| 5     | (0, 1)     | 0x60  | 0x4a  | $MFT      |
| 6     | (1, 1)     | 0x68  | 0x52  | $MFTMirr  |
| 7     | (9, 9)     | 0x60  | 0x4e  | $Quota    |
| 8     | (10,10)    | 0x60  | 0x50  | $UpCase   |
| 9     | (3, 3)     | 0x60  | 0x50  | $Volume   |
| 10    | (5, 5)     | 0x58  | 0x44  | .         |
| 11    | (0, 0)     | 0x10  | 0x00  | LAST sentinel (flags=0x02) |

(Microsoft places `$Quota` at rec 9 in modern format.com output;
ours puts `$Secure` there. The slot identity matters by name, not
by position — chkdsk identifies records by `$FILE_NAME`.)

#### Public references for the fix

- MS-FSCC `FILE_NAME` attribute value layout: parent_reference (8) +
  ctime/atime/mtime/rtime (4×8) + alloc_size (8) + real_size (8) +
  file_attributes (4) + ea/reparse (4) + name_length (1) +
  namespace (1) + name (UTF-16-LE), total 0x42 + name_len*2.
- MS-FSCC `INDEX_ROOT` attribute value: 16-byte IR header
  (attribute_type=0x30 for `$FILE_NAME` indexes, collation_rule=1
  for `COLLATION_FILE_NAME`, index_block_size, clusters_per_index)
  + 16-byte `INDEX_HEADER` (entries_offset=0x10, entries_used,
  entries_alloc, flags) + sequence of `INDEX_ENTRY`s.
- MS-FSCC `INDEX_ENTRY`: 8-byte file_reference + 2-byte
  index_entry_length + 2-byte stream_length + 4-byte flags
  (bit 1 = LAST) + variable-length stream + pad to 8.
- COLLATION_FILE_NAME (1) — case-insensitive UTF-16-LE bytewise
  compare with NTFS upcase folding. For our pure-ASCII system file
  names ($-prefix + 1..7 letters + `.`), simple ASCII upcase
  reproduces Microsoft's order exactly.

#### Justification

A category-2 diff per `corroborated-debug`'s Phase 3: "Provably
wrong against spec — one of the two values violates a documented
rule. The reference is by assumption correct; therefore ours is the
bug." Reference's `$I30` carries 12 entries; ours carries the LAST
sentinel only; spec says any record whose `$FILE_NAME.parent_reference`
points at a directory **must** appear in that directory's `$I30`.
Ours violates this; the reference does not.

#### Fix (commit `f3ea014`)

`src/mkfs.rs`:

1. Add helper `build_file_name_stream(parent_reference, nt_time,
   name, is_dir, is_system, data_alloc, data_real) -> Vec<u8>`.
   Returns the value bytes of a `$FILE_NAME` attribute (without the
   24-byte attribute header) — same byte layout `write_file_name`
   already produces in-record. Reused so each `INDEX_ENTRY`'s stream
   is byte-identical to the in-record `$FILE_NAME`.
2. Add helper `build_index_entry(file_reference, stream, is_last)
   -> Vec<u8>`. Header (16 bytes: file_ref + e_len + s_len + flags)
   + stream + pad to 8. `is_last=true` produces the LAST sentinel
   (zero-length stream, flags=0x02, e_len=0x10).
3. Add helper `build_populated_index_root_attr(attr_id,
   index_block_size, entries_blob) -> Vec<u8>`. Same outer wrapper
   as `build_empty_index_root_attr`, but with INDEX_HEADER's
   entries_used/entries_alloc set to `0x10 + entries_blob.len()`
   and the entries blob copied in after the header.
4. Add helper `collate_file_name(a, b) -> Ordering`. ASCII upcase +
   UTF-16-LE bytewise. Justified by the per-entry decode above:
   our 12 system-file names ($-prefix + ASCII alpha + `.`) sort
   identically under ASCII upcase as under NTFS upcase.
5. Restructure `format_filesystem`: declare a `Vec<(u32,
   &'static str, bool, u64, u64)>` `sys_entries` near the top of
   the rec-building region. Each rec 0..11 build (except rec 5)
   pushes its tuple at the end of its block. The rec 5 build is
   moved to AFTER rec 11 — deliberately last — so we have every
   system record's `(rec_num, name, is_dir, alloc, real)` before
   constructing root's `$I30`.
6. The relocated rec 5 block: pushes `(rec::ROOT, ".", true, 0, 0)`
   into `sys_entries`, sorts by `collate_file_name(a.1, b.1)`,
   builds `entries_blob` by streaming `build_index_entry(...)`
   results for each tuple plus a final LAST sentinel, and passes
   `build_populated_index_root_attr(3, 4096, &entries_blob)` as
   the sole attribute to `build_system_record(..., rec::ROOT, ".",
   true, 0, 0, &[index_root])`.

The per-entry stream is byte-identical to the in-record one because
both go through `build_file_name_stream`. parent_reference for every
entry is `encode_file_reference(rec::ROOT as u64, 5)` (matches the
convention in `build_system_record`). sequence per entry is
`max(1, rec_num)` — matches the iter11 fix already in place.

`$TMPDIR/rust-fs-ntfs-diag/agent-5442-2026-05-02/iter-20260502-025958`:

### iter14-v1 (this session) — pre-fill MFT rec 12-15 with minimal placeholders. REVERTED.

#### Hypothesis

After iter13 cleared the orphan messages, the remaining
`frs.cxx:60f` could be caused by chkdsk's MFT iteration finding
zero-bytes at slots 12-15 where Microsoft writes FILE-magic
placeholders.

#### Byte-diff evidence

Reference's first 16 MFT records all carry FILE magic; ours has
FILE magic only on records 0..11. Reference records 11-15 share a
template:

| Field            | reference rec 11..15            |
|------------------|---------------------------------|
| FILE magic       | yes                             |
| flags            | 0x0001 (IN_USE)                 |
| link_count       | 0                               |
| `$FILE_NAME`     | absent                          |
| `$STD_INFO`      | resident, 0x48 (short form)     |
| `$50` SD         | resident, 0x80 (104-byte SD)    |
| `$DATA`          | resident, 0x18 (empty value)    |
| END_MARKER       | yes, bytes_used = 0x130         |

#### Code change attempted

Added a `build_reserved_record(layout, record_number)` helper that
writes:

- FILE magic
- USA at 0x30, USA count = sectors+1
- sequence = max(1, record_number) per iter11
- link_count = 0
- attrs_offset = 0x48
- flags = 0x0001
- bytes_allocated = record_size
- next_attr_id = 3
- mft_rec_num = record_number
- USA value at 0x30 = 1
- `$STD_INFO` (long form, 0x60 — what `write_standard_information`
  emits)
- END_MARKER (8 bytes)
- bytes_used set to cursor

Loop after rec 11: `for rsv in [12, 13, 14, 15] { build_reserved_record;
place_record }`. Added 12, 13, 14, 15 to the MFT internal bitmap.

#### Result

Diag dir not preserved (run was reverted before commit). chkdsk
Stage 1 reported:

> File record segment C is corrupt.
> File record segment D is corrupt.
> File record segment E is corrupt.
> File record segment F is corrupt.

Records C/D/E/F = 12/13/14/15. **Regression**: Stage 1 was clean
under iter13 alone; my placeholder records broke it.

#### Why it failed (proposed)

Reference's placeholders carry `$STD_INFO` (short form 0x48) +
`$50` (Windows SECURITY_DESCRIPTOR, 0x80) + `$DATA` (resident
empty, 0x18) + END_MARKER. Mine carried only `$STD_INFO` (long
form 0x60) + END_MARKER. chkdsk likely requires `$DATA` (or `$50`)
for an in-use record, and the long-form `$STD_INFO` without a
companion SD — when no `$Secure` cache is functional — may also
trip a check.

Reverted via `git checkout src/mkfs.rs` before commit. Linux tests
green again.

#### Recovery

Iter14-v1's evidence (reference's exact placeholder template) is
preserved in this document for the next agent. Re-attempting
iter14-v1 with the reference's full attribute set requires
generating a valid 104-byte Windows `SECURITY_DESCRIPTOR` — bigger
than this session's byte budget. See "What's still broken → iter15
ranked candidates" below.

### iter14-v3 (this session) — rec 11 left empty; confirms `$Extend` is NOT the cause of `frs.cxx:60f`

This is `agent-c5fe`'s nomenclature; their iter14-v1 was a
`SECURITY_DESCRIPTOR` attempt that broke mount, their iter14-v2
restructured it to mount cleanly but didn't help. They identified
two remaining hypotheses for `frs.cxx:60f` at the close of their
iter14-v2 entry: `$Extend` at rec 11, and `$Secure`'s missing
view-index attributes.

This session's iter14-v3 tests the first.

#### Hypothesis

Microsoft's reference rec 11 has no `$FILE_NAME` (link_count=0) and
is not present in root's `$I30`. Ours had `$Extend` at rec 11 with
a `$FILE_NAME` pointing to root (parent=(5,5)) and an entry in
root's `$I30` (added by iter13). If chkdsk's reconnect-scan has
hardcoded knowledge that rec 11 is reserved-empty, having a
directory there with a parent-ref to root could trigger an internal
assertion.

#### Byte-diff evidence

| Field                  | reference rec 11             | ours pre-iter14-v3   | ours post-iter14-v3 |
|------------------------|------------------------------|----------------------|---------------------|
| FILE magic             | present                      | present              | absent (zero-bytes) |
| flags                  | 0x0001 (IN_USE)              | 0x0003 (IN_USE\|DIR) | n/a                 |
| link_count             | 0                            | 1                    | n/a                 |
| `$FILE_NAME`           | absent                       | `$Extend` → root     | absent              |
| `$INDEX_ROOT '$I30'`   | absent                       | empty `$I30`         | absent              |
| In MFT internal bitmap | yes                          | yes                  | no                  |
| In root `$I30`         | no                           | yes (12 entries)     | no (11 entries)     |

#### Code change (commits `1135519`, `6ecf58c`)

`src/mkfs.rs`:

1. Drop the `// record 11: $Extend (empty directory)` block in full.
   No record at rec 11 means the slot stays zero-bytes (`mft_buf` is
   zero-init, `place_record` is never called for rec::EXTEND).
2. Drop `rec::EXTEND` from the MFT internal bitmap list (line 240ish
   in `format_filesystem`'s rec 0 build).
3. Drop the `pub const EXTEND: u32 = 11;` constant from the `rec`
   module — no callers remain.
4. Drop the `build_empty_index_root_attr` helper — no callers
   remain (rec 5 uses `build_populated_index_root_attr` since
   iter13).
5. On `agent/agent-840e-2026-05-02` (commit `6ecf58c`), apply the
   same change to a slightly different code shape: that branch uses
   a `RootIdxEntry` struct instead of a tuple, and an explicit list
   of entries rather than the sys_entries.push flow. Drops the
   `RootIdxEntry { rec_num: rec::EXTEND, ... }` literal.

#### Result

Diag dir
`$TMPDIR/rust-fs-ntfs-diag/agent-5442-2026-05-02/iter-20260502-072921`
(after a 60-s sleep retry to clear a transient mount collision):

> Stage 2: ... 66 index entries processed.  Index verification completed.
> CHKDSK is scanning unindexed files for reconnect to their original directory.

#### What this confirms

### Pipeline parameterization (commit `bb91904`)

#### What was broken

`scripts/test-windows-local.sh` always ran with the default
`-VolumeSizeMb 256 -WrapperSizeMb 384 -Label "CITEST"` and no
cluster size override. The matrix work-list contains 23 scenarios
with different `volume_params` (32 MiB to 1 GiB; cluster 512 to
65536; labels including empty, ASCII, Latin-1, CJK, max-length
ASCII). Without parameterization, only the default one could run.

#### Code change

`scripts/run-windows-test.ps1`:

```diff
 param(
     [int]$VolumeSizeMb = 256,
     [int]$WrapperSizeMb = 384,
-    [string]$Label = "CITEST"
+    [string]$Label = "CITEST",
+    [int]$ClusterSize = 4096
 )

-./target/release/mkfs_ntfs.exe -L $Label --serial deadbeefcafe1234 nfs.img |
+./target/release/mkfs_ntfs.exe --cluster-size $ClusterSize -L $Label --serial deadbeefcafe1234 nfs.img |
     Tee-Object diag/mkfs-output.txt
```

`scripts/test-windows-local.sh`:

```bash
# New env reads near the top of the script
SCENARIO_SIZE_MB="${SCENARIO_SIZE_MB:-256}"
SCENARIO_CLUSTER="${SCENARIO_CLUSTER:-4096}"
SCENARIO_LABEL="${SCENARIO_LABEL:-CITEST}"
SCENARIO_WRAPPER_MB="${SCENARIO_WRAPPER_MB:-$((SCENARIO_SIZE_MB + 128))}"

# Forward to PowerShell with -ParamName values
ssh ${SSH_OPTS:-} "${VM_HOST}" "Set-Location '${VM_WORKDIR_PS}'; \
  powershell -ExecutionPolicy Bypass \
    -File '.\\scripts\\run-windows-test.ps1' \
    -VolumeSizeMb ${SCENARIO_SIZE_MB} \
    -WrapperSizeMb ${SCENARIO_WRAPPER_MB} \
    -ClusterSize ${SCENARIO_CLUSTER} \
    -Label '${SCENARIO_LABEL}'"
```

The wrapper auto-defaults to size + 128 MiB to preserve the
iter5-era invariant that the wrapper VHDX must hold the volume plus
GPT slack.

#### Justification

Required by every non-default scenario in the matrix. Defaults
preserved so existing iteration loops behave identically. No
behavioural change for the canonical `mac-format-basic-256mib`
path.

## What's still broken — iter15+ ranked candidates

After iter14 ($SD on every system record by sibling agents) and
this session's iter14-v3 (rec 11 empty), 17 scenarios remain at the
`frs.cxx:60f` ceiling. The remaining hypotheses, ranked:

### 1. `$Secure`'s missing view-index attributes (`$SDS`/`$SDH`/`$SII`)

**Highest yield.** By elimination: c5fe ruled out missing root SD;
this session ruled out `$Extend` at rec 11. `$Secure` is the next
candidate that's been hinted at across multiple iterations (iter12
already established that chkdsk has hardcoded knowledge of `$Secure`
and demands the `IS_VIEW_INDEX` flag; we set the flag but ship empty
view-index payloads).

What reference's `$Secure` carries:

- **`$SDS`** — *Security Descriptor Stream*. Unnamed `$DATA` stream
  containing concatenated SDs, each prefixed by a 20-byte
  `SECURITY_DESCRIPTOR_HEADER` (hash + security_id + offset +
  length).
- **`$SDH`** — Named `$INDEX_ROOT`/`$INDEX_ALLOCATION` keyed on
  (hash, security_id), values point to `$SDS` offsets. Lets
  Windows look up "is this SD already cached?" by hash.
- **`$SII`** — Named `$INDEX_ROOT`/`$INDEX_ALLOCATION` keyed on
  security_id alone, values point to `$SDS` offsets. Lets
  Windows look up "what's SD #N?" by integer.

Public references:

- MS-FSCC `$Secure` system file description (security descriptor
  caching mechanism).
- MS-DTYP `SECURITY_DESCRIPTOR` layout (for the SDs that go into
  `$SDS`).

What a minimum viable `$Secure` could carry:

- `$SDS` with a single descriptor (the same default 72-byte SD
  used by `$STANDARD_INFORMATION.security_id=0` referenced by every
  file). Header: hash (computed over the SD bytes), security_id=0,
  offset=0, length=0x48.
- `$SDH` index with one entry mapping (hash, 0) → 0.
- `$SII` index with one entry mapping 0 → 0.

This is meaningful work — three new attributes per record, two
named index attributes that may not fit resident, hash computation
matching whatever Microsoft uses (probably MD4 per Windows
Internals, but worth verifying by extracting reference's `$SDS`
header and reading the hash field). Estimate ~300-500 LOC.

If `$Secure` view indexes alone don't clear `frs.cxx:60f`, the next
move is to dump reference's full `$Secure` record's bytes and replay
them as a const blob. That's the corroborated-debug fallback — use
observed reference bytes when the spec is silent.

### 2. `$UpCase` content mismatch

chkdsk has been emitting "Read-only chkdsk found bad on-disk
uppercase table - using system table." since iter6. Treated as a
warning, not a fatal — chkdsk uses its built-in table when ours
disagrees. **Probably independent of `frs.cxx:60f`** but lower
confidence; if reconnect-scan's case-insensitive comparisons fall
back to chkdsk's table while the rest of the FS uses ours, an
inconsistency could conceivably trip an internal assertion.

The fix is to ship the canonical NT5+ `$UpCase` as a const byte
array generated independently from MS-FSCC. 128 KiB blob; fixed.
Sibling agents have referenced this in `NEXT_PLAN.md` as a known
correctness hazard.

### 3. Rec 11..15 pre-fill (with full attribute set)

Tested this session as "STD_INFO + END_MARKER only" (iter14-v1) and
chkdsk reported the records corrupt. Reference's template is
STD_INFO + `$50` (104-byte SD) + `$DATA` (empty resident) +
END_MARKER. Generating the SD bytes is the same work as candidate
#1; once that's in place, the rec 11..15 pre-fill becomes a
~30-line addition reusing the same SD const. Worth re-attempting
after iter15-`$Secure`.

### 4. Cluster-size-specific bugs (out of scope here, noted)

Sibling agent `agent-c6a1` identified four distinct cluster-size
bugs in the matrix:

- 512 cluster: mount refusal
- 1k cluster: corrupt MFT
- 8k cluster: bad attribute
- 64k cluster: rec 5 incorrect

Each is its own investigation and not affected by `frs.cxx:60f`
work. Different code paths likely involved (BPB encoding of
clusters_per_mft_record below/above cluster size, partial-cluster
boundary handling).
