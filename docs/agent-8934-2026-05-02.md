## 3. Fix #1 — iter13: root `$I30` indexes every system MFT record

### Symptom

Every system file (records 0..11) was reported as orphaned by
chkdsk's Stage 2 "scanning unindexed files for reconnect to their
original directory" sub-phase. See the `Detected orphaned file ...`
listing in §2 above.

### Diagnostic

The pipeline dumps 16 × 4 KiB MFT records from both our `mkfs_ntfs`
output and the parallel Microsoft `format.com` reference VHDX into
`diag/ours-mft-16recs.bin` and `diag/reference-mft-16recs.bin`. I
decoded record 5 (root) from each and compared the resident
`$INDEX_ROOT:$I30` attribute byte-for-byte.

**Reference root rec 5's `$INDEX_ROOT`** (1128-byte value):
- IR header (16 bytes): attribute_type=0x30 ($FILE_NAME),
  collation=1 (COLLATION_FILE_NAME), index_block_size=0x1000,
  blocks_per_index_block=1
- INDEX_HEADER (16 bytes): first_entry_off=0x10, total=0x458,
  alloc=0x458, flags=0
- 11 sorted entries, each pointing at a system record:

  | name        | mft_ref | seq | alloc/real | flags | namespace |
  |-------------|---------|-----|------------|-------|-----------|
  | `$AttrDef`  | 0x4     | 4   | 0/0        | 0x0   | 3         |
  | `$BadClus`  | 0x8     | 8   | 0/0        | 0x0   | 3         |
  | `$Bitmap`   | 0x6     | 6   | 0/0        | 0x0   | 3         |
  | `$Boot`     | 0x7     | 7   | 0/0        | 0x0   | 3         |
  | `$LogFile`  | 0x2     | 2   | 0/0        | 0x0   | 3         |
  | `$MFT`      | 0x0     | 1   | 0x10000/0x10000 | **0x6** | 3 |
  | `$MFTMirr`  | 0x1     | 1   | 0/0        | 0x0   | 3         |
  | `$Quota`    | 0x9     | 9   | 0/0        | 0x0   | 3         |
  | `$UpCase`   | 0xa     | 10  | 0/0        | 0x0   | 3         |
  | `$Volume`   | 0x3     | 3   | 0/0        | 0x0   | 3         |
  | `.`         | 0x5     | 5   | 0/0        | 0x0   | 3         |

  Plus a 16-byte END sentinel (eflags=0x02).

- Order is the NTFS COLLATION_FILE_NAME upcase order; for ASCII
  names it reduces to byte-sorted.
- Reference's modern format puts `$Secure` under `\$Extend\$Secure`
  (a view-index host inside `\$Extend`, not in root). At rec 9
  Microsoft's modern `format.com` reuses the historic `$Secure` slot
  for `$Quota`. Our layout (legacy NTFS 3.0 era) puts `$Secure`
  directly at rec 9 and `$Extend` at rec 11.

**Our root rec 5's `$INDEX_ROOT`** (48-byte value, before iter13):
- IR header + INDEX_HEADER as above
- ZERO real entries — only the END sentinel.

### Root cause

`src/mkfs.rs`'s `build_empty_index_root_attr` constructed an
$INDEX_ROOT attribute that held only the END sentinel. For an empty
directory that's correct; for the root directory it's wrong because
NTFS requires every directly-named file's $FILE_NAME to be findable
through its parent directory's `$I30` index — and ALL system files
are direct children of root.

### Public references consulted

- Microsoft MS-FSCC structure references for the on-disk `$INDEX_ROOT`
  layout (IR_HEADER + INDEX_HEADER + entries + END sentinel).
- Microsoft MS-FSCC structure for the `$FILE_NAME` attribute (used
  as the index entry key).
- The per-system-file MFT reservation conventions from publicly
  documented NTFS layout (record numbers 0..11 reserved for the 12
  system files).
- Iteration 11 of `docs/chkdsk-findings.md` — confirmed that
  `parent_reference` and the per-record sequence_number on system
  records must match the value the `$I30` index entries use, with
  `seq = max(1, rec_num)` for system files.

NO Linux-NTFS-implementation source was consulted. The byte-diff
against Microsoft's own `format.com` output is the corroboration.

### Fix

`src/mkfs.rs` commit `7e87e87`:

1. Refactored `build_empty_index_root_attr` →
   `build_index_root_attr(attr_id, index_block_size,
   bytes_per_sector, entries: &[RootIdxEntry])`. An empty slice
   produces the END-sentinel-only output (preserves $Extend's empty
   directory behaviour); a non-empty slice emits each entry in
   order, padded to 8-byte alignment, then the END sentinel.

2. Defined a new struct:
   ```rust
   struct RootIdxEntry<'a> {
       rec_num: u32,
       seq: u16,
       name: &'a str,
       alloc: u64,
       real: u64,
       fn_flags: u32,
   }
   ```
   The function emits, per entry: 8-byte mft_ref (rec_num | seq<<48)
   + 16-byte entry header + a $FILE_NAME structure as the key:
   parent_ref=(root=5, seq=5), zeroed timestamps, alloc/real from
   the entry, fn_flags, namelen + namespace=3 (WIN32_DOS) + UTF-16
   name + padding to 8-byte align.

3. At the rec 5 call site I built a hand-sorted array of 12 entries
   (our 12 system files; reference has 11 because it lacks $Extend
   and uses $Quota at rec 9 instead of our $Secure). Sort order is
   the NTFS upcase / byte-sorted: `$AttrDef`, `$BadClus`,
   `$Bitmap`, `$Boot`, **`$Extend`**, `$LogFile`, `$MFT`,
   `$MFTMirr`, **`$Secure`**, `$UpCase`, `$Volume`, `.`.

4. `$MFT`'s entry mirrors the standalone $FILE_NAME on rec 0 per
   iter9: alloc=real=`mft_clusters * cluster_size`, flags=0x06
   (FILE_ATTRIBUTE_HIDDEN | FILE_ATTRIBUTE_SYSTEM). Every other
   entry uses 0/0/0 to match reference behaviour.

5. Updated `tests/mkfs_roundtrip.rs::format_and_parse_back`. The
   prior assertion `assert!(names.is_empty())` was asserting the
   buggy behaviour; replaced with
   `assert_eq!(names, expected)` where `expected` is the 12-name
   sorted list. Per `dev-loop`'s "exception: the test was asserting
   buggy behavior that you intentionally fixed" rule, the test
   change is called out explicitly in the commit.

### Verification

Linux: 7/7 mkfs tests passing (lib + smoke + roundtrip).
Lint: `cargo fmt --check`, `cargo clippy --all-targets -- -D warnings`
clean.
VM (`mac-format-small-64mib`, diag dir `iter-20260502-030330`):

- Pre-fix: 12 `Detected orphaned file <name>` lines, frs.cxx 60F
  after.
- Post-fix: 0 orphan lines, Stage 1 + Stage 2 both reach
  `Index verification completed`, frs.cxx 60F still after.
- /scan exit 11 → 3 (3 = elevation-required for /scan in our
  unprivileged SSH; the meaningful exit is read-only chkdsk's,
  still 3 because of the residual frs.cxx 60F).

### Result

Fix lands as commit `7e87e87`. The orphan-recovery family of errors
is fully eliminated; chkdsk now gets all the way through Stage 2's
index verification. The remaining frs.cxx 60F is a separate bug
that iter14 then attacked.

```
Header (20 bytes):
  rev=1, sbz1=0, ctrl=0x8004 (SE_SELF_RELATIVE | SE_DACL_PRESENT)
  owner_off=72, group_off=88, sacl_off=0, dacl_off=20
```

Mask `0x00120089` =
`SYNCHRONIZE | READ_CONTROL | FILE_READ_DATA | FILE_READ_EA | FILE_READ_ATTRIBUTES`
— effective read-only access. Same SID for Owner, Group, and one of
the ACEs is BUILTIN\Administrators. The other ACE is NT AUTHORITY\SYSTEM.
This matches the standard "system file" SD used on NT-era system
records.

While decoding the SD I also compared `$STANDARD_INFORMATION`:

| | reference rec 0 $SI | ours pre-iter14 rec 0 $SI |
|---|---|---|
| value bytes | 48 | 72 |
| `file_attributes` (bytes 32..36) | `0x06` (HIDDEN \| SYSTEM) | `0x26` (HIDDEN \| SYSTEM \| ARCHIVE) |
| `max_versions`/`version_number`/`class_id` (bytes 36..48) | 0/0/0 | 0/0/0 |
| `owner_id`/`security_id`/`quota_charged`/`update_seq_number` (bytes 48..72) | **NOT PRESENT** | 0/**0**/0/0 |

Reference uses the 48-byte pre-NTFS-3.0 compact form. Ours used the
NTFS-3.0+ 72-byte extended form. The extended form claims a
`security_id` that points into `$Secure`'s `$SDH`/`$SII` indexes;
we set it to 0, which is invalid because `$Secure` has no entry 0.

This is the most likely cause of `frs.cxx 60F`: chkdsk's "scanning
unindexed files" sub-phase, while walking each MFT record, reaches
the security-id validation step, fails to find entry 0 in our empty
`$Secure`, and trips the generic sanity check at frs.cxx line 0x60F.

### iter14c (DISPROVEN): rec 11 (`$Extend`) as flat file, no `$FILE_NAME`, removed from root's $I30

#### Hypothesis

Reference's rec 11 is structured as `0x10/0x50/0x80` (a flat file
with empty `$DATA`, NO `$FILE_NAME`, flags=0x1 / no IS_DIRECTORY).
Reference also doesn't put `$Extend` in root's `$I30`. Maybe
chkdsk's "scanning unindexed files" trips on our directory-form

## 7. Deferred bugs (still failing in the matrix)

Beyond `frs.cxx 60F` (which all `mac:format` scenarios at default
cluster size hit), the matrix surfaced two cluster-size-specific
bugs that need separate fixes.

### iter15 — corrupt MFT at cluster_size < 4096

#### Symptom

```
Stage 1: Examining basic file system structure ...
Corrupt master file table. Windows will attempt to recover
master file table from disk.
Windows cannot recover master file table.  CHKDSK aborted.
```

#### Affected scenarios

- `mac-format-cluster-512` (cluster_size=512)
- `mac-format-cluster-1k` (cluster_size=1024)

#### Diagnostic

`mkfs_ntfs` reports a successful format, the wrapper VHDX mounts
fine and gets a drive letter, but chkdsk's first MFT read fails.
Format succeeds = boot sector is good enough; chkdsk fails = MFT
records aren't where chkdsk expects.

#### Hypothesis

`mft_record_size` is fixed at 4096 (default). At cluster_size=1024,
an MFT record spans 4 clusters. The boot sector encodes
`clusters_per_record` as `cpmr = mft_record_size / cluster_size`
when mft_record_size ≥ cluster_size (positive small number); per
publicly documented NTFS layout `cpmr` is `i8`. Possibilities:

1. `cpmr` is encoded with the wrong sign or magnitude at this
   cluster size.
2. MFT records are written at the wrong byte offset because the
   placement loop assumes 1 cluster per record but should advance
   by `mft_record_size / cluster_size` clusters per record.
3. `$MFT`'s own `$DATA` data-run encoding misuses the cluster
   count (allocation should still be `mft_clusters * cluster_size`
   = 256 KiB, but the run-length might encode wrongly).

`src/mkfs.rs:654` (around line 654 per iter15 entry in the findings
doc) has the conditional that selects the `cpmr` encoding. Inspect
that first.

### iter15b — `$BadClus:$Bad` corrupt at cluster_size = 8192

#### Symptom

```
Stage 1: Examining basic file system structure ...
Attribute record (80, $Bad) from file record segment 8 is corrupt.
Errors found.  CHKDSK cannot continue in read-only mode.
```

#### Affected scenarios

- `mac-format-cluster-8k` (cluster_size=8192)

Cluster sizes 4096 and 65536 do NOT show this; only 8192 does. So
this is a specific edge case in run-length encoding at this
cluster count.

#### Diagnostic

`$BadClus`'s named `$Bad` `$DATA` attribute is built non-resident
as a sparse run covering `cluster_count` clusters at LCN=None. At
8192-byte clusters, a 256 MiB volume has 32K clusters (= 0x8000).
At 4096-byte clusters it has 64K (= 0x10000). At 65536-byte
clusters it has 4K (= 0x1000).

The data-run length encoding (per public NTFS layout) packs the
length into the smallest power-of-two byte count needed. 0x8000
needs 2 bytes (signed) but only just; perhaps `encode_runs` picks
1 byte (off-by-one boundary).

#### Hypothesis

`src/data_runs.rs::encode_runs` has an edge case at 32K-cluster
sparse runs. The fix is to review against publicly documented
NTFS run-length encoding and verify every cluster-count value
(power of 2 from 1024 up to 1M+) round-trips correctly.

#### iter14e (most likely): root SD with inheritance ACEs

The biggest remaining structural diff at the byte level. Build a
new const `ROOT_SD: [u8; ~248]` modelled after reference's root SD

- 8 ACEs in the same order/shape (inheritable pairs for Admins,

- Owner = S-1-5-32-544
- Group = some generic SID (reference uses a machine-specific

to use `ROOT_SD` instead of `SYSTEM_SD`:

```rust
let sd_blob: &[u8] = if record_number == rec::ROOT {
    &ROOT_SD
} else {
    &SYSTEM_SD

let sd_attr = build_resident_unnamed(ATTR_SECURITY_DESCRIPTOR, 2, sd_blob);
```

If chkdsk's frs.cxx 60F clears, this was the cause. If not, move on

Our 32-byte resident bitmap is spec-valid but reference uses

Lower likelihood of being the cause — both forms are spec-valid
