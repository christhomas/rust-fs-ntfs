## Cycle 5 — iter14 + iter15 attempts at the frs.cxx 60f assert

After the user supplied a working SSH key and asked me to "continue
to work on the issues according to the plan," I attacked the
trailing `frs.cxx 60f` assert directly with two evidence-corroborated
layout fixes:

### iter14 — `$SECURITY_DESCRIPTOR` (0x50) on every system MFT record

Branch commit `c0fde08`. Reference's first 16 MFT records all carry
a `$SECURITY_DESCRIPTOR` attribute (104 bytes for non-root system
files, 248 bytes for root); ours had **none**. Three unique SD blobs
captured byte-for-byte from `format.com` reference output and baked
in as `SD_SYSFILE_RO` / `SD_SYSFILE_RW` / `SD_ROOT_DIR` constants.
`build_system_record` writes the SD attribute between `$FILE_NAME`
and the caller's extra attrs (preserving canonical attr-type order).

Result: SD presence verified per-record on disk. **Targeted
`frs.cxx 60f` assert PERSISTS unchanged.** Layout fix is real and
spec-corroborated, but does not address the assert.

### iter15 — FILE-magic placeholders for unused MFT slots

Branch commit `4ee3bad`. Reference's MFT (16 records) carries FILE
magic in every slot — slots 12-15 are minimal 304-byte placeholders
with `seq=N`, `flags=0x01` IN_USE. Ours pre-allocated 64 MFT slots
but only 12 had FILE magic; slots 12-63 were raw zeros. chkdsk
reports "64 file records processed" so it iterates the whole MFT.

Fix: after writing the 12 system records, loop `slot in 12..mft_records_capacity`
and write a FILE-magic placeholder (FILE magic + seq=0 + IN_USE
CLEAR + just header + end marker). IN_USE CLEAR is correct because
our `$MFT:$Bitmap` keeps bits 12+ clear (those slots are genuinely
free for user files; reference happens to reserve 12-15 as system).

Result: placeholders confirmed on disk (`FILE seq=0 flags=0x0
used=80 attrs_off=0x48 end=0xffffffff` per slot 12-15). **Targeted
`frs.cxx 60f` assert PERSISTS unchanged.** Layout fix is real and
spec-corroborated, but does not address the assert.

### What this rules out

Five evidence-corroborated layout fixes have landed since iter10
(iter11/12 on local main, iter13/14/15 on this branch). The
`frs.cxx 60f` assert has survived every one of them. **Strong
indication the cause is content-level not layout-level.** The
remaining hypotheses, in priority order:

1. **`$UpCase` content** (highest priority — also fixes the
   "bad on-disk uppercase table" warning). Our
   `src/upcase.rs::generate_upcase_table()` synthesises a mapping
   from `char::to_uppercase()`; chkdsk wants the canonical NT 3.x
   table byte-for-byte. chkdsk uses `$UpCase` for filename collation
   in the orphan-recovery scan — mismatched table → confused
   comparison → very plausibly the assert source. **Next-step
   recipe**: patch `run-windows-test.ps1` to copy
   `clusters[upcase_lcn..upcase_lcn+32]` from a `format.com`-formatted
   reference VHDX into `diag/reference-upcase.bin`; bake those 128 KiB
   into a const that replaces the current generator output.
2. `$LogFile` content (we fill with 0xFF; format.com initialises with
   proper RSTR-led LogFile records).
3. `$MFT:$Bitmap` non-resident value (worth a byte-diff vs reference).
4. `$AttrDef` blob byte-diff.

All four candidates need the existing pipeline patched to dump those
specific clusters/files into `diag/`. The pipeline currently dumps
only MFT records and boot sectors.
